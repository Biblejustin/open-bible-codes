# Greek Expanded Prospective Exact-Center Preregistration

Status: prospective candidate-discovery screen.

This document freezes the next Greek exact-center screen before any new outputs
from that screen are inspected. It is designed to find a review queue, not to
promote claims.

## Question

Do any new declared Greek theological, biblical, apocalyptic, festival, tribe,
or Table-of-Nations terms produce same-skip phrase extensions that:

1. have exact-center surface context;
2. appear by exact extension key in one or more Greek NT source labels;
3. can be sorted by source-version distribution for later controls?

## Term List

Locked term file:

- `terms/greek_expanded_prospective_terms.csv`

The term list is generated by:

```bash
python3 -m scripts.build_greek_expanded_prospective_terms
```

Fixed generation rules:

- source files:
  - `terms/theological_terms.csv`
  - `terms/greek_nt_claim_terms.csv`
  - `terms/prophetic_terms.csv`
  - `terms/biblical_festivals.csv`
  - `terms/biblical_tribes.csv`
  - `terms/table_of_nations.csv`
- Greek rows only;
- normalized length at least 4;
- dedupe by normalized Greek form;
- exclude every normalized form already present in
  `terms/greek_exact_center_cohort_terms.csv`.

This intentionally excludes the prior exact-center cohort, including `δοξα`,
so this run does not recycle the strongest known row.

Compared Greek NT source labels:

source is data, not automatic failure.

## Protocol

Run:

```bash
python3 -m scripts.run_protocol protocols/greek_expanded_prospective_exact_center.toml --resume
```

Locked search settings:

- corpora: TR_NT, BYZ_NT, TCG_NT, SBLGNT;
- skip range: `2..50`;
- direction: both;
- minimum term length: 4;
- surface-context hits only;
- extension lookback/lookahead: 12 letters;
- phrase lexicon length: up to 4 words;
- include both-sided extensions;
- max extensions per hit: 20;
- extension summary filters:
  - minimum extension length: 3;
  - minimum base-term length: 4;
  - match kind prefix: `phrase_`;
  - top rows per corpus: 3000.

## Candidate Routing

This first protocol does not run controls. It routes candidates:

- all-source rows: strongest control queue;
- multi-source rows: source-distribution review queue;
- source-only rows: source-specific review queue;
- BYZ_NT + TCG_NT only: related-source caution queue.

Rows that survive this screen still need:

- paired controls;
- context review;
- letter-path audit;
- correction across tested rows;
- synthetic/null baseline comparison.

## Promotion Boundary

No row in this screen may be labeled:

- `claim`
- `confirmed_code`
- `proof`
- `prophecy`
- `statistical discovery`

The maximum status from this run alone is:

- `prospective_review_queue_candidate`

## Interpretation Boundary

This screen can identify new prospective review rows. It cannot establish
meaning by itself. Hidden-path-only rows are normal ELS candidates, not
failures. A same-span surface echo is stronger but is not required for entry
into the review queue.
